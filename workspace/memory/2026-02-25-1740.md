### Puntos de Acoplamiento (Coupling)

| Acoplamiento | Severidad | Detalle |
|--------------|-----------|---------|
| **UI ↔ Export JSON** | HIGH | Board page lee workitems.json directamente; cualquier cambio en schema del motor rompe UI |
| **runScript ↔ Exec** | HIGH | Usa `exec` (no spawn); no streaming real; timeout fijo 30s; potencial zombie processes |
| **Logs ↔ Filesystem** | MED | Paths hardcoded con `join()`; no hay abstracción de storage; race conditions posibles en concurrent writes |
| **Actions ↔ Secuencia** | MED | `executeActionWithExport` hace 1+N llamadas sincrónicas; rollbacks manuales en cada endpoint |
| **UI Monolito** | MED | Board page ~900 líneas; lógica de estado, DnD, modales, formularios todo junto |

### Single Points of Failure

1. **`wi-export.sh`**: Si falla, todas las mutaciones retornan error (export post-action es obligatorio)
2. **`runScript.ts` whitelist**: Si un script necesario no está en `WHITELISTED_COMMANDS`, acción queda bloqueada irreversiblemente
3. **RUNNING.json**: No hay mecanismo de "stale run" (crash deja RUNNING.json huérfano indefinidamente)
4. **Concurrent writes**: Múltiples requests simultáneos a mismo ID pueden corromper logs (no hay file locking)

### Framework Reusable vs Específico Brokia

| Componente | Naturaleza | Notas |
|------------|------------|-------|
| `src/components/ui/*` | **REUSABLE** | Botones, Cards, Modales — no dependen de dominio Brokia |
| `src/components/board/*` | **ESPECÍFICO** | DndKit + lógica Kanban + workitem types |
| `src/server/runScript.ts` | **HYBRID** | Whitelist es específico, pero patrón exec+log es genérico |
| `src/server/runs.ts` | **REUSABLE** | Patrón RUNNING.json puede servir para cualquier job queue |
| `src/server/logs.ts` | **ESPECÍFICO** | Acoplado a estructura de directorios Brokia |
| Contrato API | **ESPECÍFICO** | blocked_by_guardrail + stdout/stderr es diseño Brokia |
