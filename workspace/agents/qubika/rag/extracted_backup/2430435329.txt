Setup de un proyecto local
Para el setup local de cualquier proyecto dockerizado seguiremosestedocumento.

Branches
Cuando desarrollamos en bancard, siempre lo hacemos en una branch propia de esa tarea que luego mergearemos a develop. Siempre la nombraremos en base al ticket de Jira. ej:PRYPOR-697.Para crearla, se recomienda usar el comandogit checkout -b nombre-de-la-branch.La flag-bcrea una branch a partir de la que estamos parados y nos muevedirectamente a la misma, todo en el mismo comando. Es importante estar parados endevelopal hacer esto y antes ejecutar ungit pullpara asegurarnos que la branch “madre” esté actualizada.

Ej:

Tips: Para que ya quede la branch en el repositorio al cual luego vamos a subir nuestros cambios, es recomendable ejecutar el comandogit push --set-upstream origin nombre-de-la-branch.Esto es para que luego al momento de pushear un commit no tengamos que preocuparnos de que la branch en la que trabajamos se encuentre o no en el repositorio de GitHub.

Cambios en el proyecto
Siempre que realicemos cambios en un proyecto local deberemos tener tests para ellos. Utilizaremos RSpec como framework para la creación de estos tests y FactoryGirl para la representación de las entidades de los modelos. Podés leer la documentación básica de RSpecaquíasí como también verestapresentación sobre la forma de testear que manejamos en bancard.

Cuidado: Podés guiarte con los tests ya hechos en los proyectos, pero por lo general estos están desactualizados ya que la sintaxis que se usó quedó deprecada. Más info enestapresentación. También se recomienda usar como guía de tests PR’s recientes de compañeros.

Ejecutar tests
Para ejecutar los tests localmente, y como estamos en un ambiente dockerizado, debemos primero acceder al contenedor en donde está corriendo el proyecto que queremos testear. Para ello en la terminal utilizaremos el comandodocker exec -it nombre-del-contenedor bash. Una vez dentro, procedemos a correr los tests utilizando el comandobundle exec rspec.

Tips:
Para ver los contenedores que están corriendo locamente utilizamos el comandodocker ps. De ahí podremos sacar el nombre del contenedor.

Si quieres probar un test en específico y para no tener que correr todos los tests (demora un tiempo), podemos agregarle al comando anterior lo siguiente:bundle exec rspec ruta-del-test. Esto ejecutará únicamente el archivo de test que le indicamos con la ruta.

Ejemplo:bundle exec rspec spec/controllers/movements_controller_spec.rb
También podemos agregar al anterior comando la opción de solo ejecutar una línea de código específica dentro del archivo de test. Para ellos utilizamos:bundle exec rspec ruta-absoluta-del-test:linea-de-código.

Ejemplo:bundle exec rspec .spec/controllers/movements_controller_spec.rb:56

Crear Commit
Una vez que nuestros archivos modificados se encuentren en lastaging area,creamos el commit utilizando el comandogit commit -m “mensaje-descriptivo-del-commit.”Respecto al mensaje, seguiremosestaguía de estilos. Como convención dentro de bancard, siempre el mensaje del commit tendrá antes el código de la tarea entre paréntesis rectos. Ej:git commit -m “[PRYPOR-697] - mensaje-descriptivo-en-inglés”

Una vez hayamos creado el commit con su respectivo mensaje procederemos a pushear nuestros cambios a la branch que creamos anteriormente. Para ello, utilizaremos el comandogit push origin nombre-de-nuestra-branch.Una vez ejecutados estos comandos tendremos nuestros cambios pusheados en la branch remota.

Esto indicará el nombre de nuestra branch, hace cuanto se pushearon nuestros cambios y el botón verde el cual nos permitirá crear la PR.

Si no llegásemos a ver este cartel, nos dirigimos a la branch que creamos anteriormente o directamente al link que nos devuelve GitHub al crear y pushear nuestra branch (ver Branches).

Procederemos a clickear en el botón “Compare & pull request” y esto es lo que mostrará:

Lo primero y más importante es la barra de encima donde leemosbase: developycompare: nombre-de-nuestra-branch.Aquí debemos asegurarnos que la PR siempre sea contra develop y comparando desde nuestra branch la cual tiene el commit con todos los cambios. A la derecha en el apartado ‘Reviewers’ siempre agregaremos a todos nuestros compañeros del team, que serán notificados para luego hacercode-review. En la caja de texto opcionalmente podríamos agregar una breve descripción del cambio, simplemente para aportar al contexto de la PR. Luego de modificar estos pasos, procedemos a clickear en “Create pull request” y la PR se creará.

Code Review
Luego de que nuestros cambios se encuentran en una PR y agregamos como reviewers a todos nuestros compañeros, llega el momento en que los demás revisan nuestros cambios. A esta corrección grupal la llamamos code review. Es trabajo de todos revisar y ser meticulosos a la hora de ver código ajeno ya que esto es crucial para evitar posibles futuros errores. Es normal que siempre haya detalles para corregir, ya sean mejoras en la lógica de nuestros cambios o errores de tipeo (typos). Los comentarios se hacen en inglés y siempre hay lugar para discutir sobre los mismos. Cuando un compañero llega a la conclusión de que el código está OK, se obtiene unapprove.

Tip:

En bancard al momento de agregar un cambio a una PR, lo que utilizaremos será la siguiente secuencia de comandos:

Luego de efectuar los cambios, los agregamos al staging area congit add .

Seguido de esto, utilizaremos el comandogit commit --amend.Este comando actualiza el último commit sin necesidad de deshacerlo ni volver a poner el mismo mensaje. Es importante tener los cambios stageados antes de utilizar este comando. Al hacerlo, se abrirá una ventana vim. Para salir de ella y guardar tipeamos:wq(write and quit)

Por último debemos pushear este commit ahora actualizado con las correcciones. Para ello volveremos a utilizar nuestro querido git push pero agregando al final la flag--force-with-leasela cual sobreescribirá los commits de la branch solamente si los cambios que tenemos localmente son los únicos en esa branch.

Como resultado final ahora el commit que se encontraba en la PR está actualizado respecto a los comentarios de nuestros compañeros. Al utilizar la flag-fsobreescribimos el commit viejo con el nuevo.

Merge
Luego de recibir por lo menos 2 approves y que los tests en circle CI se ejecuten sin errores, podremos mergear nuestros cambios a develop. Para esto simplemente bajamos hasta el final de la PR y seleccionamos la opción a utilizar para el merge. Aquí veremos 2 opciones:Rebase and merge,Squash and merge y Create a merge commit(ignoraremos las opciones deCreate a merge commit y Squash and mergeya que no acostumbramos a usar esta opción en Bancard).

Seleccionaremos la opción deRebase and merge. Esta es la opción a elegir ya que siempre preferimos tener todos los cambios en un solo commit.

Una vez elegida la opción nuestro/s commit/s se mergearan a la branch contra la que hicimos nuestra PR.

Deploy
Finalmente, es hora de deployar nuestros cambios. Previamente habremos mergeado los mismos a una branch cómodevelop.Existe la posibilidad de que nuestra iniciativa tenga una branch aparte en la cual ya existan otros cambios y querramos agregar los nuestros ahí. Por lo general estas siempre se nombran comofeature/nombre-de-la-iniciativa o provide/nombre-de-la-iniciativa.
Ej:feature/preauthorizations.

Una vez que nuestro commit fué mergeado a la branch contra la que hicimos la PR,  nos dirigimos hacia el repositorio del proyecto y, seleccionando la branch a la cual mergeamos nuestros cambios, veremos que hay un apartado por encima de los archivos del proyecto en donde se muestra el último commit.

Ej:

Como vemos en la imagen, el último commit es “[PRYPOR] - View for preauthorizations report” y podemos notar un tick verde a la izquierda del commit hash. Esto indica que la imagen de este proyecto fue buildeada correctamente por circle CI. Al momento de mergear, veremos en lugar de este tick un punto amarillo que indica que el buildeado de la imagen está en proceso. Deberemos esperar a que se muestre el tick verde para continuar con el deploy.

Una vez que la imagen del proyecto está buildeada, deployaremos. Para esto, primero debemos entrar al bastión mediante ssh utilizando este comando:
sshsoporte_ais02@10.100.14.4

Una vez dentro del bastión entraremos al servidor de Deploy, también mediante ssh, utilizando este comando:
sshservicios@10.100.14.57

Una vez dentro del servidor de deploy, ingresamos a la carpeta de bancard-docker, la cual se encuentra en/opt/ruby/bancard-docker, utilizando el comandocdpara movernos entre los directorios. Existe también un link simbólico desde el home hacia esta ruta, por lo cual podemos movernos directamente hacia ella utilizando el comandocd bancard-docker. Dentro de la carpetatiramosel deploy. El comando tipo para el mismo es el siguiente:

./bin/bancard-deploy -p <nombre-del-proyecto> -e <enviroment> -t <nombre de la imagen>

-p <nombre-del-proyecto> ejs: bancard-billing-service, bancard-commerce-service, bancard-commerce-suite, etc

-e <environment> por defecto siempre es staging, se agregue o no se agregue la flag.

-t <nombre de la imagen> será el nombre de la branch la cual sebuildeóla imagen. Si mergeamos a develop, el nombre será ‘develop’. Sino, será el nombre de la branch sin elfeature/oprovide/

Recomendamos leer el README de bancard-docker para información más detallada respecto al comando usado para el deploy. Lo puedes encontraraquí.

Ej:

Una vez confirmada la configuración, el deploy comenzará a ejecutarse. Una vez terminada la ejecución, nuestro trabajo quedará deployado y el flujo de trabajo finalizado.

Bancard-stubs
Dentro del espectro de gemas de Bancard,bancard-services-stubses la más utilizada en los proyectos al momento de escribir los stubs en nuestros archivos de test. Aclarar en primer lugar que no es necesario usar esta gema para realizar un stub, pero hay muchos proyectos que la utilizan y por eso debemos conocer su funcionamiento.

Dentro del repositorio hay 2 ramas: La principal, que es la más nueva y que se sigue actualizando, se llamarefactor(la que está seleccionada por defecto). También estámasterque contiene una versión más vieja de la gema pero que aún se utiliza en algunos proyectos. Estas coexisten como dos versiones muy diferentes de la misma gema. Notaremos cual de las dos usa nuestro proyecto ya que estará indicado bajo la key ‘branch’ en el Gemfile.

Notamos que se definen 2 gemas, llamando al mismo repositorio. ‘bancard-services-stubs’ es la que apunta a la versión vieja, la rama master, mientras que ‘bancard-stubs’ apunta a la versión nueva, la rama refactor. El ‘require: false’ de estas líneas sirve para decirle albundlerque instale la gema pero no la deje como requerida para todo el proyecto, ya que esto consumiría recursos innecesariamente.
