Dudas
Angus-SDOC: Se puede usar una representation como root de la respuesta? O siempre que se use el atributo type en una response dentro de operations.yml tiene que ser anidado a un atributo padre?

Los messages.yml exigen "level", "status_code". Estos se usan para exponer la docu. En la API Rails hay que renderizar errores haciendo que coincida con messages.yml.

Que pasa si devuelvo un error que no tengo declarado en operations? No hay problema con eso, simplemente tu cliente no cuenta con que puedas devolver ese message en ese endpoint. angus-remote funciona bien.

Formato del json al devolver error. para los errores rescatados con rescue_from se puede usar un template. Y para los 500?

Que pasa con los 500? Hay que formatearlos? Habría que rescatarlos y formatearlos para que angus-remote lo pueda entender. (principalmente keys status y mesages)

La doc en html expone la ruta /rails-blogs-api/api/1.0, es decir tiene ese prefijo hardcodeado ("/#{@service.code_name}/api/#{@service.version}#{operation.path}"). La doc en JSON no agrega ese prefijo. angus-remote al usar el JSON, usa la ruta que le configures. Meter la API en rails para que este dentro de esa ruta base?

Angus no responde otro status code que no sea 200 para éxito. Devolver 200 en rails aunque la semántica de la operación amerite otro código? Ver que pasa si se responde 2XX con angus remote a ver si no se queja. 201 created, 204 no content, etc. angus-remote no se fija en el status code, se puede devolver cualquier cosa. Excepto 500, 501 y 503 los cuales angus-remote trata de manera especial y lanza excepcion.

Versionado de la API y los docs?

angus-remote no funciona en ruby 3.2.2 porque se elimina el metodo File.exists? (usar File.exist?) Funciona en 3.1.4, no se en que version deja de funcionar.

Angus remote parece que tiene el metodo .status_code pero que no funciona. Para fijarse si una request anduvo bien o mal, se fija en la key "status" del body. Y se puede hacer .status. por ejemplo:

Definitivamente angus-remote trata de forma especial a la key "status" del body de la respuesta, seteandolo como una variable de instancia @status.
Desventaja de angus: en los distintos {resource}/operations.yml no se pueden repetir claves de operaciones. Por ejemplo no puedo tener el show en mas de un recurso, ya que después se usa remote_service.show. En su lugar hay que tener show_blog, show_comment, etc.

Al renderizar un error, la key messages del body es esperado que sea un array de objetos con las keys level, key y dsc. (level y key deberían coincidir con messages.yml). Como si pudiera haber varios errores en una misma API call. Generalmente es una sola excepción. ver casos de uso de eso y si hay solución. Imagino que para errores de validación no seria muy complicado. Update: ya lo implementé para RecordInvalid.

Mantener sincronizados los codigos de respuesta de los errores (404, 422, etc) de la implementación en Rails con los de los messages

angus-remote no se fija en la semantica de los codigos de respuesta. Un 204 con body vacío tras eliminar un recurso por ejemplo.
